> How do I configure this project in Supabase so I am amble to do step 2?       

● Here's the exact click-by-click walkthrough:

  ---
  1 — Create a Supabase account

  Go to supabase.com → click Start your project → sign in with GitHub (or email)

  ---
  2 — Create a new project

  1. Once logged in, click "New project"
  2. Fill in:
    - Name: locked-in-factory (or anything you want)
    - Database Password: create a strong password and save it somewhere — you'll
   need it if you ever connect directly to the DB
    - Region: pick the one closest to you (e.g. East US)
  3. Click "Create new project"
  4. Wait ~1–2 minutes for it to provision (you'll see a loading screen)

  ---
  3 — Get your credentials

  Once the project is ready:

  1. In the left sidebar click "Project Settings" (gear icon at the bottom)
  2. Click "API" in the settings submenu
  3. You'll see two things you need:

  Project URL:        https://abcdefghijklmnop.supabase.co
  anon public key:    eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJpc3MiOiJzdX...

  4. Copy both values and paste them into js/supabase-config.js:

  const SUPABASE_URL      = 'https://abcdefghijklmnop.supabase.co';
  const SUPABASE_ANON_KEY = 'eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...';

  ---
  4 — Disable email confirmation (for local dev)

  1. In the left sidebar click "Authentication"
  2. Click "Providers" — confirm Email is toggled ON (it is by default)
  3. Click "Settings" (still inside Authentication)
  4. Find "Enable email confirmations" → toggle it OFF
  5. Click Save

  This means users can sign up and immediately access the site without needing
  to verify their email. Re-enable this before publishing.

  ---

● Here's exactly what changed and why:

  The problem: storage.js used raw keys like sp_subjects — the same key for
  every user.

  The fix — two parts:

  1. storage.js — synchronous prefix resolution

  At load time, _resolveUserPrefix() reads the Supabase session directly from
  localStorage and extracts the user ID. Every key is then namespaced:
  sp_subjects → abc123_sp_subjects. This happens synchronously before any page
  JS runs, so there's no timing issue.

  2. auth.js — reload on login instead of hiding overlay in-place

  After a successful login/signup, window.location.reload() is called. This
  triggers a clean page load where storage.js picks up the new session and
  resolves the correct prefix from the start — no stale data, no race
  conditions.

  Result:
  - User A logs in → sees userA_sp_subjects, userA_sp_sessions, etc.
  - User A logs out, User B logs in → page reloads → sees userB_sp_subjects,
  userB_sp_sessions, etc.
  - Neither user ever sees the other's data
